# Single-Agent Systems

## What Is an Agent?

In the context of large language models, the word "agent" has a precise meaning: an agent is an LLM that has been given access to tools. A tool is any function the model can call to fetch data, run calculations, or interact with external systems. Without tools, an LLM is limited to generating text from its training data. With tools, it can look up live information, execute code, query databases, and take actions in the real world. The combination of language understanding and tool execution is what transforms a passive text generator into an active problem solver.

The simplest possible agent is a single LLM operating under a single system prompt, equipped with one or more tools, and responsible for an entire workflow end to end. This is the single-agent pattern, and it is the natural starting point for any agentic application. Before reaching for multi-agent frameworks or autonomous planning loops, it is worth understanding how much a single well-prompted agent can accomplish on its own.

## The Single-Agent Pattern

A single-agent system uses one system prompt to define the agent's identity and capabilities, then directs it through multiple tasks by varying only the user prompt. The agent does not change personas or hand off work to other agents. It simply wears multiple hats under one coherent identity.

The accompanying notebook, `SingleAgent_DB`, demonstrates this pattern with a practical business application. A single "Data Analyst" agent receives an uploaded Excel file and performs three distinct tasks: designing a logical database schema from the spreadsheet's structure, generating SQL DDL statements to create the corresponding tables, and writing SQL queries to answer natural-language business questions. All three tasks share the same system prompt. The agent's behavior changes not because its identity shifts, but because the user prompt describes a different job to be done.

This design has significant practical advantages. There is only one system prompt to write, test, and maintain. Each business query requires a single LLM call rather than the multiple calls a multi-agent system would need. The result is faster response times, lower API costs, and simpler debugging. When something goes wrong, there is exactly one place to look: the single agent's reasoning on that particular prompt.

## When a Generalist Agent Is Sufficient

For many real-world applications, a single-agent approach is not merely adequate but preferable. When the tasks in a workflow are closely related, such as reading data, designing a schema, and writing queries against that schema, a single agent with broad instructions can handle them all competently. Modern LLMs are remarkably good generalists. The same model that designs a database schema can write the SQL to populate it and then query it, because these tasks draw on overlapping knowledge about data structures and SQL syntax.

Single-agent systems are the right choice when speed and cost matter more than cross-validation, when the workflow is straightforward enough that a single chain of reasoning can handle it, and when the primary goal is to get correct answers rather than to audit the reasoning process from multiple perspectives. In the `SingleAgent_DB` notebook, students can observe that the SQL queries generated by one generalist agent are often nearly identical to those produced by a dedicated "SQL Developer" agent in the multi-agent version of the same workflow.

## When Specialization Is Needed

The single-agent pattern reaches its limits when the workflow demands genuinely different expertise at different stages, when cross-checking between independent approaches adds meaningful value, or when the complexity of the system prompt grows so large that the agent loses focus. If a single prompt tries to make the agent simultaneously an expert database architect, a SQL optimization specialist, a natural-language analyst, and a quality evaluator, the prompt becomes unwieldy and the agent's performance on each individual task may degrade.

The honest lesson from comparing single-agent and multi-agent approaches on the same problem is that specialization often matters less than practitioners expect. A common and effective pattern is to start with a single agent, observe where it fails, and add specialized agents only to address those specific failure modes. Building five agents upfront because a framework encourages it is a form of premature optimization that adds complexity without guaranteed benefit.

The next chapter explores what happens when you do split work across multiple agents, and critically examines when that added complexity is justified by measurable improvements in output quality.

## Key Takeaways

:::{admonition} Key Takeaways
:class: tip
- An agent is an LLM with access to tools; a single-agent system uses one system prompt and one LLM to handle an entire workflow by varying only the user prompt.
- Single-agent systems are faster, cheaper, and simpler to debug than multi-agent alternatives because each query requires only one LLM call and there is exactly one place to look when something goes wrong.
- The practical lesson is to start with a single agent, identify specific failure modes, and add specialized agents only to address those failures --- not because a framework encourages it.
:::

## Exercises

**Easy:** Build a single-agent system that takes a CSV file, describes the columns, and generates three SQL queries to answer business questions. Use one system prompt for all tasks.

**Medium:** Test the same agent on two different datasets (e.g., sales data vs. HR data) without changing the system prompt. Document where the generalist prompt succeeds and where it struggles with domain-specific nuances.

**Challenge:** Deliberately push the single-agent pattern to its limits by giving it a five-step workflow (e.g., schema design, DDL generation, data validation, query writing, and result interpretation). Identify the point at which the system prompt becomes too complex and performance degrades, then propose where you would split into multiple agents.

For exploring what happens when you decompose a workflow across multiple specialized agents with cross-validation, see **Ch 15 Multi-Agent**.
